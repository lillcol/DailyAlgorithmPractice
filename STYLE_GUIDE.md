# 全局格式规范（统一风格）

## 文档规范

- 全部使用 Markdown。
- 标题层级固定：`#`（题名）-> `##`（大章节）-> `###`（小节）。
- 示例统一写法：
  - `输入：...`
  - `输出：...`
  - `解释：...`
- 复杂度统一写法：
  - `时间复杂度：O(...)`
  - `空间复杂度：O(...)`

## 插图与示意图（可选）

- **推荐做法**：在每题目录下建 `images/`，把图片放进该目录，在 `README.md` 里用**相对路径**引用，并把图片一并提交到 Git。
  - 示例：`![双指针示意](./images/two-pointers.png)`
  - 好处：离线可用、GitHub / Gitee 网页预览、换机器克隆后路径仍然正确。
- **若本地预览不显示**：先确认图片文件确实存在且文件名大小写一致；Cursor / VS Code 的 Markdown 预览对**相对路径**最稳，避免用绝对路径（如 `/Users/...`）。
- **不想存二进制图时**：
  - 简单流程可用 **Mermaid** 代码块（多数平台支持渲染）；
  - 极简结构可用 **ASCII 示意图**；
  - 或外链图（依赖网络与外链稳定性，长期归档不如本地 `images/`）。

## 命名规范

- 题目目录名：`YYYY-MM-DD-leetcode-编号-题目短名`（如 `2026-03-26-leetcode-3-无重复字符最长子串`）
- Java 类名：`SolutionLC<题号>`（如 `SolutionLC3`）
- Java 方法名：语义化驼峰命名（如 `twoSum`）
- Python 函数名：snake_case（如 `two_sum`）

## 注释规范

- 注释重点解释“为什么”，不是重复“做了什么”。
- 关键分支、边界处理、复杂度关键点必须写注释。
- 不写过度冗余注释（逐行翻译式注释）。

## 代码顺序规范

1. `SolutionLC<题号>.java`（先写）
2. `SolutionLC<题号>.py`（后写）

## README 内容顺序（每题）

1. 题目信息（标题、难度、题型、日期）
2. 题目描述
3. 输入输出示例
4. 边界条件
5. 多解法（一般解法 / 经典解法 / 最优解法）
6. 通俗讲解
7. 复杂度分析（至少对一般解法与最优解法给出）
8. 代码链接（Java / Python）

## 解法要求（每题）

- 如果题目存在明显分层解法，必须写出：一般解法、经典解法、最优解法。
- 每种解法至少说明：核心思路、适用场景、时间复杂度、空间复杂度。
- 代码可只实现最优解法，但 README 必须保留多解法对比结论。
